### 域名选择：4条黄金法则

#### **简短优先原则**

1. 理想长度在6\-14个字符之间。短域名便于用户记忆和输入。

    1. 如 `notion\.so` 就比 `notion\-productivity\-app\.com` 好很多。

#### **域名后缀选择**

1. 优先选择以下后缀的域名

    1. `\.com` 仍然是全球认可度最高的后缀；

    2. `\.ai`是做AI工具站的首选后缀，因为搜索引擎会天然的把\.ai网站视作具有AI性质的网站；

    3. 如果目标市场是特定国家，可考虑国别域名如 `\.cn`、`\.``co\.uk`，但对于新手做国际市场，优先抢 `\.com`；

    4. `\.org` 是做公益性质网站的标准后缀，如果你希望让用户认为你的网站纯公益且免费，可以考虑

#### 雷区勿进

1. 请尽量避免以下雷区

    1. 连字符（\-）

    2. 数字

    3. 双写字母（如 pressstart 里的 ss）

    4. 容易拼错的单词

#### 关键词与品牌的微妙融合

1. 过去\&\#34;精确匹配域名\&\#34;（EMD）如 `bestrunningshoes\.com` 有搜索加成，但Google现在更看重内容质量

    1. 建议是：如果关键词能自然融入品牌名（如 `canva\.com` 暗示 canvas），很好；但不要为了塞关键词牺牲品牌感。

### 路径 A：截图喂 AI

#### Step 1 · 收集参考截图

1. 去拆 pollo\.ai / viddo\.ai，截下：首页、模型详情、Pricing页、生成结果页

    1. 📝Tips：也可以在mobbin\.com找更多参考网站，在Mobbin搜索 AI video generator就可以找到很多参考案例

2. 用飞书截屏，可以滚动屏幕截图，可以把页面完整截图

#### Step 2 · 和AI讨论设计方案

1. 把截图发给AI，把这段prompt发给它，和它讨论确定设计方案。

2. 讨论后，最终AI会输出一份详细的设计文档。后续的前端实现就按照这份设计文档来

```Shell
你是一位拥有 15 年经验的工业级产品设计师，曾在 Apple Human Interface
  团队工作。你的任务是和我（一名独立开发者）一起，把我对一款产品的「模糊设计想法」变成一份**Apple 工业级精度**的 design-tokens 设计文档。

  我会给你：
  1. 5-10 张参考截图（来自我喜欢的 AI 视频聚合站 / 创作类工具）
  2. 一段我对产品的初步描述

  你不要立刻输出 design-tokens。先和我**完成一轮深度需求对齐**。请按以下顺序追问我，每次只问一组（不要一次问完，让我有思考节奏）：

  ---

  【第 1 组 · 产品边界与功能范围】
  1.1 这个产品的核心使用场景是什么？用户从打开到完成第一次成功的路径是什么样？
  1.2 关键页面有哪些？请按用户旅程列出（例：Home / Generate / Pricing / Model Detail / Template / History / Account）
  1.3 哪些页面是 SEO 落地页（来流量）？哪些是产品内页（留住用户）？两类页面的设计语言可能不同，需要分别对待
  1.4 是否需要响应式？移动端访问占比预估？是 mobile-first 还是 desktop-first？

  【第 2 组 · 用户人群】
  2.1 主要面向哪些国家？（美国 / 欧洲 / 东南亚 / 拉美 / 全球）—— 不同地域审美差异巨大
  2.2 是否有侧重的年龄段？（Gen Z 18-25 / 千禧一代 26-40 / X 世代 40+）
  2.3 是否有侧重的职业群体？（创作者 / 营销人员 / 电商卖家 / 学生 / 设计师 / 普通消费者 / B 端企业用户）
  2.4 用户的设备偏好？（高端 Apple 生态 / Windows 主流 / 移动为主）
  2.5 用户对 AI 的认知水平？（重度玩家 vs 第一次用 AI 的人）—— 影响信息密度和引导设计
  2.6 用户的付费心智？（愿意为效率付费 vs 喜欢免费玩 vs 企业账户）—— 影响 Pricing 视觉权重

  【第 3 组 · 设计偏好（感受层面）】
  3.1 看完我给的 5-10 张截图，你最想抓住哪 3 个共性？为什么？
  3.2 你希望产品给用户**第一秒**的感受是什么？请用 3-5 个形容词描述（例：clean / playful / professional / cinematic / mysterious / warm / futuristic /
  honest）
  3.3 你最不希望的感受是什么？（例：cheap / AI slop / template-like / over-designed / cold）
  3.4 在以下风格倾向中，你更接近哪一种？请按权重打分（总分 100）：
     - Apple 极简（大量留白 / 高对比 / 系统字体）
     - Linear / Vercel 工程美学（精确 / 暗色 / 等距字体点缀）
     - Krea / Runway 创作工具（深色 + 霓虹渐变 / 沉浸式）
     - Notion / Stripe 文档级清晰（友好 / 圆润 / 信息层级清楚）
     - Y2K / Brutalism 反叛（不规则 / 高饱和 / 故意粗糙）
     - Editorial / 杂志级（衬线大字 / 强排版 / 摄影感）
     - Glassmorphism 玻璃质感（半透明 / 模糊 / 高光）
     - 其他（请描述）
  3.5 主色你有偏好吗？或者希望从截图里提炼？
  3.6 你希望整体偏暗色 / 浅色 / 双主题（深浅自动切换）？
  3.7 字体你有偏好吗？（Inter / Geist / SF Pro / Söhne / Söehne Mono / Newsreader / 自定义 webfont / 不在意）
  3.8 你的产品 logo 已经定了吗？如果有，发给我（影响视觉系统）

  【第 4 组 · 技术与实现约束】
  4.1 你打算用什么前端栈？（Next.js + Tailwind + shadcn/ui ? 还是别的？）
  4.2 你能否接受用付费 webfont（影响首屏加载）？
  4.3 你能否接受复杂动效（Framer Motion / GSAP）？
  4.4 你能否接受暗色模式作为默认（产品视觉冲击更强但欧美 SaaS 多浅色）？
  4.5 你的目标 Lighthouse Performance 分？（>90 vs >70）—— 影响是否能用大图大动效

  ---

  我每一组提出后，你回答完，我都会**复述确认 + 追问 1-2 个细节**，确保我们彻底对齐。
  四组问题全部完成后，你说「OK 我准备好了，请生成 design-tokens.md」我才输出最终文档。

  最终输出的 design-tokens.md 必须达到 **Apple 工业级**精度，结构如下：

  # design-tokens.md

  ## 0. 设计哲学（300-500 字）
  说明这套设计的灵魂、为什么是这样、什么样的用户会被吸引、什么样的用户会被劝退。
  回答：「为什么不是 Pollo / Viddo / Heyvid 那样？」

  ## 1. 设计原则（5-7 条）
  每一条原则给出：
  - 原则名称
  - 一句话定义
  - 正面示例（这样做 ✓）
  - 反面示例（不这样做 ✗）

  ## 2. 色彩系统
  2.1 主色（primary / primary-hover / primary-active / primary-disabled，给 hex + HSL + 在 OKLCH 色彩空间的值）
  2.2 辅色（secondary / accent，同上）
  2.3 语义色（success / warning / error / info，每种 5 个梯度）
  2.4 中性色（neutral 50-950 共 11 个梯度）
  2.5 表面色（surface / surface-elevated / surface-overlay）
  2.6 文本色（text-primary / text-secondary / text-tertiary / text-disabled / text-on-color）
  2.7 边框色（border-default / border-subtle / border-strong / border-focus）
  2.8 暗色模式映射表（每个 token 给出暗色版本）
  2.9 色彩使用规则（什么场景用什么色，禁忌）
  2.10 给出可直接粘贴的 Tailwind config 片段

  ## 3. 字体系统
  3.1 字体族（display / body / mono，每种说明用途 + fallback 链）
  3.2 字号梯度（10 级，从 caption 到 display-xl，给出 px / rem / line-height / letter-spacing）
  3.3 字重规范（regular 400 / medium 500 / semibold 600 / bold 700 的使用场景）
  3.4 标题层级（H1-H6 的字号 / 字重 / 行高 / 上下间距）
  3.5 段落规范（最长字符数 / 段间距 / 列表缩进）
  3.6 数字字体规则（金额 / 倒计时 / 数据展示是否用 mono / tabular-nums）
  3.7 Tailwind config 片段

  ## 4. 间距系统
  4.1 基础单位（4px / 8px 网格选哪个，为什么）
  4.2 间距梯度（0 / 1 / 2 / 3 / 4 / 6 / 8 / 12 / 16 / 24 / 32 / 48 / 64 / 96 共 14 级）
  4.3 组件内间距规范
  4.4 组件间间距规范
  4.5 页面布局栅格（max-width / gutter / column）
  4.6 移动端断点（sm / md / lg / xl / 2xl）

  ## 5. 圆角系统
  5.1 圆角梯度（none / sm / md / lg / xl / 2xl / full）
  5.2 使用规则（按钮 / 卡片 / 输入框 / 头像 / 模态框 各用什么）

  ## 6. 阴影系统

  6.2 每级用途（卡片 / 悬浮 / 模态框 / 抽屉 / 焦点环）
  6.3 暗色模式下的阴影策略

  ## 7. 动效系统
  7.1 缓动函数（ease-in / ease-out / ease-in-out / spring，给 cubic-bezier 值）
  7.2 动效时长（instant 0 / fast 150ms / normal 200ms / slow 300ms / slower 500ms）
  7.3 标准过渡组合（hover / press / appear / disappear / slide-in / slide-out）

### 第 4 步 · 视频生成管线设计（聚合站灵魂章节）

这一章是 AI 视频聚合站的核心，必须详细展开。

#### 4.1 同步 vs 异步决策

视频生成 5-60 秒，**必须异步**。给出 3 种实现路径，按复杂度递增：

**路径 A · Polling（最简单，MVP 推荐）**
- 前端 POST /generate → 后端调 fal.ai → 立即返回 job_id
- 前端每 2 秒 GET /status/:job_id 直到完成
- 适合：DAU < 500，并发 < 20
- 优点：实现 30 分钟，零基础设施
- 缺点：前端轮询费流量，单实例容量受限

**路径 B · Server-Sent Events（中等）**
- 前端建 SSE 连接，后端推送进度
- 适合：500 < DAU < 5000
- 优点：实时性好，省流量
- 缺点：长连接管理复杂，Vercel Serverless 不友好

**路径 C · Queue + Webhook（规模化）**
- 用 Inngest / Trigger.dev / Upstash QStash 做任务队列
- fal.ai 完成后 webhook 回调更新状态
- 前端 SSE 或 WebSocket 接收完成通知
- 适合：DAU > 5000
- 优点：可观测、可重试、可扩容
- 缺点：架构最复杂

**给学员的建议**：MVP 走路径 A，500 DAU 后升级到路径 C。

#### 4.2 Provider Adapter Pattern（必做）

聚合站灵魂：抽象一个 `VideoProvider` 接口，下面挂 fal.ai / Replicate / 直连官方 API 三个实现。**接第二个模型时不用改业务逻辑**。

给出代码骨架（TypeScript / Python 任选一种）。

#### 4.3 模型路由策略

用户感知不到底层模型差异，按用户选「快速 / 标准 / 电影」三档路由到不同模型。给出路由表示例：

```
快速（5 Credits/秒）  → Seedance 2 Fast ($0.022/s)
标准（15 Credits/秒） → Kling 2.5 Turbo ($0.07/s)
电影（25 Credits/秒） → Veo 3.1 Fast ($0.10/s)
```

每秒 Credits 价格 = API 成本 × 2-3 倍（毛利空间）

#### 4.4 失败兜底

- API 超时（>60s）→ 自动取消 + 退还 Credits
- API 5xx → 重试 1 次 → 失败则降级到备用 Provider
- 内容审核失败 → 不扣 Credits，提示用户修改 prompt
- Credits 已扣但生成失败 → **必须自动退**（用户体验红线）

#### 4.5 并发控制

每个用户限制最大并行任务数：
- Free 用户 1 个
- Lite 2 个
- Pro 3 个
- Premium 5 个

防止单用户拖垮 API quota。

#### 5.2 视频存储成本估算

按用户量分级算：

- 早期（50 GB）：R2 $0.75/月
